## Бизнес-правила: что уже перенесено и что ещё нет

### Перенесено на фронт/моки
- Валидация `SUM(participants.share_amount) == total_amount`.
- Минимум 1 участник с `is_payer = true`.
- Минимум 2 участника в расходе.
- Правило округления равного деления: остаток копеек добавляется первому участнику.
- Пересчёт долгов после create/update/delete расхода.

### Пока не закрыто полностью
- Нет серверной транзакционности и атомарности пересчёта (`BEGIN/COMMIT`) — только клиентские моки.
- Нет полноценного ACL уровня спеки (`is_admin`, 403/404 по членству, запреты удаления участника/группы при долгах).
- Нет полного контрактного API-слоя (`handler/service/repository`) и реальных HTTP-ошибок/кодов.
- Нет ручного UI для произвольных долей участников (сейчас деление автоматическое равномерное).
- Маршруты деталей расхода в UI пока не вложены в `/groups/:id/...`.

## Сравнение: фронт сейчас vs целевой backend

| Область | Фронт сейчас | Целевая модель в спеке |
|---|---|---|
| Оплата | Погашение сделано через `balance` в мок-сервисе | Погашение через реальный endpoint `POST /groups/:id/balances/:bid/pay` |
| Долги | Есть persisted-слой `balancesStore` в памяти | Персистенс в PostgreSQL + транзакционный пересчёт |
| Расходы | `participants[]` + инварианты суммы и плательщика реализованы | Те же инварианты, но сервер валидирует и хранит |
| Контроль доступа | Базовый UI-контроль | Строгий ACL и ошибки доступа на сервере |
| Контракты API | Структуры данных приближены к спеке | Полное совпадение request/response и status codes |
