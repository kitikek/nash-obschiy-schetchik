## Что сейчас отсутствует как бизнес-правила/контроль (не endpoint’ы)

- `SUM(participants.share_amount) MUST == total_amount` и правила округления/остатка — сейчас это “почти совпадает” только в рамках равного деления; явной проверки контракта share_amount как в спеках нет (`src/pages/GroupDetail/AddExpenseModal.tsx`, `src/services/expenses.ts`, `src/utils/calculateBalances.ts`).
- Контроль прав (`is_admin`, запреты на удаление участника/группы при наличии долгов) — частично отображается в UI (например, удаление участника доступно только создателю группы в `GroupDetail`), но полноценной серверной валидации и всех продуктовых ограничений из спеки нет.
- Доступ по членству в группе и возврат `403/404` — нет: роуты в `src/App.tsx` просто требуют “аутентифицированности”, но не проверяют членство в конкретной группе.

## Сравнение логики (существенные различия)

> Левый столбец `что делаем?` оставлен пустым для ваших решений/заметок.

| что делаем? | Фронт (как сейчас) | Бэк (как планировалось в `backend_spec.md`) |
|---|---|---|
|  | **Логика оплаты привязана к каждой трате:** должник в карточке расхода нажимает «Я оплатил», а плательщик вручную подтверждает этот факт. | **Логика оплаты привязана к балансу группы:** гасится долг между должником и кредитором через `balance` (частично/полностью), а не через ручные статусы внутри конкретной траты. |
|  | **Долги считаются “на лету” только для отображения:** список «кто кому должен» формируется на клиенте из текущего массива расходов. | **Долги — постоянная доменная сущность:** после каждого изменения расходов сервер пересчитывает и сохраняет `balances` транзакционно, чтобы это была единая точка правды. |
|  | **Модель расхода упрощена:** пользователь выбирает плательщика и участников, сумма делится поровну автоматически. | **Модель расхода детализирована:** у каждого участника своя доля (`share_amount`), сумма долей обязана совпадать с общей суммой, с формальными правилами округления. |
|  | **Жизненный цикл расхода неполный:** нет сценариев редактирования и удаления, значит нет “истории пересчёта” после правок. | **Жизненный цикл полный:** расход можно создать/обновить/мягко удалить, и любое действие должно запускать пересчёт долгов. |
|  | **Управление участниками в основном “добавить”:** нет продуктового сценария удаления участника и связанных ограничений. | **Управление участниками с ограничениями:** нельзя удалять создателя и нельзя удалять участника при непогашенных долгах; действия доступны только админам группы. |
|  | **Группа ведёт себя как локальный объект UI:** удаление/деактивация группы не учитывает финансовое состояние как обязательное правило. | **Группа подчиняется финансовым инвариантам:** при непогашенных долгах деактивация/удаление должно блокироваться бизнес-правилами. |
|  | **Доступ в основном по факту логина:** пользовательская навигация не обеспечивает строгую модель “видишь только свои группы” как серверное правило. | **Доступ по членству в группе:** любой доступ к чужим данным должен жёстко отсекаться (Forbidden) на стороне бэка. |
|  | **Профиль и метрики считаются локально из того, что уже есть на экране/в памяти.** | **Профиль и метрики считаются централизованно на сервере,** чтобы цифры были консистентны между всеми клиентами и с БД. |
